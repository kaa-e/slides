10.1 デットコード
どんな条件であっても決して実行されないコードをデッドコード、または到達不能コードと呼びます。
<!-- コード入れる -->
コードの可読性低下：実際には実行されないにもかかわらずどういう条件で実行されるかを読み手に考えさせてしまいます。また、なぜ実行されないようなコードが残っているのか、何か意図があるのではと、読み手を混乱させます。

将来バグになる可能性：これまで到達不能だったのが、なんらかの仕様変更によりデッドコード周辺のロジックが変わり、到達可能になる場合があります。蘇ったコードが仕様と異なっていればバグになります。

デッドコードは、発見次第すぐに削除しましょう。IDEの静的解析機能にはデッドコードを検出する機能があり、便利です。サービスやツールを活用して、デッドコードを一掃しましょう。


10.2 YAGNI原則
実際の開発においても将来の仕様を予見し、ついつい先回りしてつくり込んでしまうことがあるでしょう。しかし、先回りで実装されたロジックは、現実にはほとんど使われないばかりか、バグの原因になるなど悪魔化することが多いです。
YAGNIと呼ばれる設計原則「You aren’t going to need it.」の略。訳すと「今は必要ないでしょう」になります。実際に必要になったときにのみ実装せよ、という方針です。
ではYAGNI原則を守らず先回りしてつくると何が起こるのでしょうか？
ソフトウェアに対する要求は日々変化していきます。仕様として確定しておらず、明確に言語化もされていない要求を予測して実装しても、ほとんどの場合予測は外れます。予測が外れれば、デッドコードになります。また、あちこちにあやふやな予測が入り込むので、先回りでつくったロジックは複雑になりがちです。
可読性が低下し、読み手を混乱させます。
何かの変更をキッカケに実行されるとバグになる可能性が非常に高いです。なぜなら、先回りでつくったロジックは、現実には存在しない不確実な仕様にもとづいているからです。
先回りでつくり込んだ分だけ時間が無駄になります。今必要な機能だけをつくり、構造をシンプルにしましょう。可読性が高くなり、保守や変更が容易になります。無駄な工数がかからなくなり、もっと重要な仕事に取り組めます。


10.3 マジックナンバー
ロジック内に直接書き込まれている意図不明な数値をマジックナンバーと呼びます。
マジックナンバーは実装した本人でも、あとから読み直したときに意図を理解できなくなるケースがあります。また、同一のマジックナンバーを複数の箇所に実装すると重複コードになり、状況がさらに悪化します。たとえば、仕様変更でお試し購読の消費ポイントが60から50に変更されたら、マジックナンバーの実装箇所すべてを修正しなければなりません。修正漏れがあるとバグになります。
数値の意図を明示するために、数値に名前を付けて定数として定義しましょう。

10.5 グローバル変数
グローバル変数：どこからでもアクセス可能な変数
Javaの言語仕様にはグローバル変数はありませんが、static変数をpublic宣言することでグローバルアクセス可能となります。

どこからでも参照、操作可能な変数であるため、一見扱いやすいと思うかもしれません。しかし、実態はその逆です。 
さまざまなロジックでグローバル変数を参照し、値を変更すると、どのタイミングで値が書き換わったのか把握が困難になります。
グローバル変数を参照しているロジックに変更が入ると、同じグローバル変数を参照している他のロジックにバグが生じる可能性が高くなります。カプセル化されたロジックに比べ、かなりの注意が必要です。 　

グローバル変数は、使い方によっては排他制御が必要な場合が生じます。排他制御は慎重に設計しないと、ロック時間が長くなってパフォーマンスが低下します。排他制御の設計を誤ると、デッドロックに陥る可能性があります。 　

巨大データクラスもグローバル変数としての性質を帯びやすいです。さまざまなデータを保有しているために、多くの箇所から参照されやすいからです。さらに排他制御に関しては、巨大データクラスはグローバル変数より悪質です。排他制御したいインスタンス変数が1個であっても、ほかのインスタンス変数までロックされてしまうので、パフォーマンス上大きな問題になります。 　グローバル変数を使っていなくとも、グローバル変数と同質のものを知らず知らずの内に使っているのです。 

10.5.1　カプセル化とパッケージ依存の設計で影響範囲を小さくする 　
グローバル変数が用いられているコードでは、グローバル変数を操作するロジックがあちこちバラバラに実装されています。 　バラバラな問題を解決するのがカプセル化。データとそのデータを操作するロジックをひとつのクラスにカプセル化しましょう。 　また、そのデータを必要としている箇所がどこなのか、依存関係を洗い出しましょう。
実はそれほど依存がなく、一部のパッケージだけが依存している、ということがあります。必要なパッケージだけが依存するように設計します。こうすることで呼び出し箇所を限定でき、影響範囲が小さくなります。影響範囲が小さくなると、データの参照や更新の把握が容易になり、保守や変更が楽になります。 　グローバル変数の利用がどうしても必要なケースは多くないはずです。可能な限りグローバル変数として定義せず、カプセル化し、アクセス可能なパッケージやクラスを限定しましょう。


10.6 null問題
nullが入り込む前提でロジックを組むと、いたるところでnullチェックが必要になります。nullチェックだらけでコードの見通しが悪くなりますし、nullチェックが漏れるとバグになります。 　
nullは、適切な参照先がないことを表現します。 　しかし、商品名が未設定の状態、商品の配送先が未設定の状態など、主に未設定状態をnullとして実装しているコードが世の中に多くあります。何かを持っていない状態や未設定状態も、立派な状態なのです。 　こうした使い方は、null本来の用途とは異なります。nullの誤った使い方により、多大な損失を生みかねません。

10.6.1 nullを返さない、渡さない、代入しない
null例外によるトラブルやnullチェックを避けるために、そもそもnullが入り込まない設計にする。
具体的には次を満たす設計にします。 
・nullを返さない：メソッドの戻り値としてnullをreturnしないこと 
・nullを渡さない：メソッドの引数にnullを渡さないこと 
・nullを代入しない：変数にnullを代入しないこと 　
防具の例では、装備していない状態をnullとして表現していました。nullではなく、Equipment型のstatic finalなインスタンス変数EMPTYにします。

11章 名前設計
本章の基本的な考え方は目的駆動名前設計：ソフトウェアで達成したい目的に焦点を合わせて名前を設計（命名）。名前から目的や意図が読み取れることを重視します。

11.1　悪魔を呼び寄せる名前 　
名前設計が不十分なために悪魔を呼び寄せてしまうケース。たとえばECサイトでの商品です。ありがちなのは、商品をそのまま「商品クラス」と設計してしまうこと。 
ECサイトは商品を中心に成り立っています。出品、予約、注文、発送など、商品を取り扱うユースケースがたくさんあります。したがって商品クラスは、さまざまなユースケースのクラスと結びつきます。商品クラス自体も、結び付いたクラスに関係するロジックを持ちはじめ、どんどん巨大化複雑化していきます。 　こうして巨大化した商品クラスに対し仕様変更が発生した場合どうなるでしょうか？　変更影響によりバグが生じないか、商品クラスに関連のあるクラスすべてをチェックしなければなりません。影響範囲が広すぎるために、開発生産性が低下します。

11.1.1　関心事ごとに分割する 　
商品の周囲には、予約、注文、発送などさまざまな関心事が取り巻いています。商品クラスはさまざまな関心事と結び付き、結び付いた関心事のロジックを持ってしまっています。 　商品クラスは、第6章で解説した関心の分離にもとづき、関心事それぞれのクラスへの分割が必要です。

11.1.2　関心事にふさわしい命名 　
関心事にふさわしい命名→あとは分離したクラスそれぞれに、各関心事にふさわしいロジックをカプセル化する。
　
こうすることで、目的ごとに関心が分離できます。 関心を分離しておけば、たとえば注文に関して仕様変更が生じた場合、注文関連のクラスだけに注意を払えばよくなります。影響範囲が小さくなり、開発生産性が向上します。 

11.1.3　大雑把で意味が不明瞭な名前 　
開発初期に決めた名前は、大雑把で指し示す範囲がぼんやりしているものが多いです。大雑把だと何がマズいのでしょう？　開発現場では下記のようなことが非常に多く見受けられます。 
ex: 「商品」という名が大雑把すぎて、商品に関するあらゆるロジックが実装できそうに見えてしまいます。
大雑把で意味がガバガバな名前は、あらゆるロジックを引きつけてしまう。 
「商品」は販売目的を表現できていそうに名前に見えますが、予約、注文、発送といったさまざまな目的に使われやすい、意図があいまいな名前であることがわかります。
こうした目的不明なクラスを目的不明オブジェクト。
目的不明オブジェクトは、あっという間に巨大化してしまいます。 　こうした事態に陥らないよう、関心の分離を意識した名前を設計します。関心の分離には、業務目的を名前として表現することがポイントになります。 

11.2　名前を設計する—目的駆動名前設計 
筆者はクラスやメソッドに名付けることを、「命名」ではなく、「名前設計」と呼んでいます。ここでの設計は「ある課題を解決するためのしくみや構造を考え、つくり上げること」という意味で使います。 　
プログラミングにおける名前の役割は、可読性を高めることだけではない。 業務目的に焦点を合わせた名前を付与することは、関心の分離を実現する上で重要です。設計上大きな意味があるため、「名前を設計する」といいます。 　

目的駆動名前設計は、ソフトウェアで達成したい目的や意図を、名前から読み取れるようにします。 　目的駆動で名前を設計するポイントは次の通りです。 
・可能な限り具体的で、意味が狭い、目的に特化した名前を選ぶ 
・存在駆動ではなく、目的駆動で名前を考える 
・どんな業務目的があるか分析する ・声に出して話してみる 
・利用規約を読んでみる ・違う名前に置き換えられないか検討する 
・関心が分離されているか点検する

11.2.1　可能な限り具体的で、意味が狭い、目的に特化した名前を選ぶ 
　そして顧客向けプロダクト開発における目的は、「業務的にどういう目的を達成したいのか」という業務目的となります。 　業務目的に特化することで、以下の効果が生まれます。 
・名前とは無関係なロジックを排除しやすくなる 
・クラスが小さくなる 
・関係するクラスの数が少なくなる 
・関係クラス個数が少ないので、仕様変更時に考慮を要する影響範囲が小さく済む
・目的に特化した名前なので、どこを変更すれば良いかすぐ探し出せる 
・開発生産性が向上する 

11.2.2　存在駆動ではなく目的駆動で名前を考える 　
業務目的に特化した名前とはどんなものなのか、具体的に解説します。 　まずは、目的に特化してないケースを考えてみます。たとえば「人」や「ユーザー」といった、人物が存在することに着目した存在駆動の名前を考えてみます。こうした名前をECサイトのクラス名にするとどうなるでしょうか。ECサイトの利用者は、個人以外に法人である可能性もあります。「ユーザー」が個人と法人両方の意味を持ち、複数の関心事が混ざってロジックが混乱する可能性があります。 　

存在に着目したクラス名は、用途が多重になりがちで、目的不明オブジェクトになります。ロジックが混乱します。 　
したがって、具体的な目的が明確にわかるような、目的駆動の名前にします。

ECサイトでの住所を利用する目的は、商品の配送です。したがって、単に「住所」といった無味乾燥な存在駆動の名前ではなく、「配送元」「配送先」と目的特化の名前にすると混乱を防げます。 
「金額」は存在駆動であり、どうとでも解釈可能です。目的特化にすると、「請求金額」「消費税額」「延滞保証料」「キャンペーン割引料金」などが考えられます。 
「ユーザー名」にしても、「アカウント名」「表示名」「本名」などのように、目的に応じた命名が考えられます。 

11.2.3　どんな業務目的があるか分析する 
業務目的に特化した命名をするには、どんな業務目的があるか網羅する必要があります。そのために、ソフトウェアが対象とする目的や事柄を分析する必要があります。 　
登場人物や事柄を列挙したり、関係性を整理したり、分析してみましょう。 　チームで集まって、ホワイトボードや模造紙に書いてみるのが良いでしょう。ふせんを使えば、関係し合うものをグルーピングするなど、考えの整理がよりはかどるでしょう。 

11.2.4　声に出して話してみる
名前がたくさん書き出されても、慣れていないと名前の背後にある業務目的まで書き出されることがなかなかありません。 　名前も大事ですが、業務的にどんな目的を達成したいのか、そのためにどう使われ、何と関係するのか、なぜそうするのかの理由など、業務の視点から背景や意図を整理し、チームで理解を一致させることが重要です。 　目的や意図の認識がお互いに違っていると、名前が十分にブラッシュアップされません。お互いの理解の違いを発見し解決するには、やはり声に出して話してみることが効果的です。 
ラバーダッキングと呼ばれるデバッグ手法があります。プログラミングで何か問題が発生したときに、それを誰かに説明すると自ら原因に気づき、自己解決する手法です。声に出して話してみるのは、分析でもデバッグでも効果的なやり方なのです。 
この声に出して話す分析のやり方は、書籍『ドメイン駆動設計』（「18.1.11 エリック・エヴァンスのドメイン駆動設計」）のユビキタス言語の考え方に由来します。
ユビキタス言語とは、チーム全体で意図を共有するための言葉です。同じ意図の名前を、会話、ドキュメント、クラス名やメソッド名で共通して使うことで、意図が伝わりやすくなり、設計のいびつさを解消するのに役立ちます。チームでユビキタス言語をつくるとき、会話し続け、継続的にブラッシュアップすることの重要さをドメイン駆動設計では説いています。

11.2.5　利用規約を読んでみる 
利用規約には、サービスの取り扱いやルールが極めて厳密な言い回しで書かれており、目的に特化した名前の参考になります。

11.2.6　違う名前に置き換えられないか検討する 
せっかく選んだ名前の意味が十分に狭くなかったり、複数の意味を持っている場合がよくあります。違う名前に置き換えてみて、意味をもっと狭くできないか、違和感がないかなど検討してみましょう。

11.2.7　関心が分離されているか点検する 　
目的に特化した名前を選ぶと、その目的とは関係の弱いロジックを寄せ付けにくくします。目的に強く関係するロジックが集まりやすくなり、関心が分離されます。関係の弱いロジックが混入しそうならば、名前を見直しましょう。 　また、関連しているクラスの数を確認しましょう。先に挙げた商品クラスのように、何個も関連付けられているのは良くない兆候です。もっと狭い意味の、目的に特化した名前を探してみましょう。複数の意味を持っているなら分解しましょう。関連するクラスが少なければ少ないほど、変更の影響範囲は小さくなります。

11.3　設計時の注意すべきリスク
11.3.1　名前無頓着になるな 　
目的駆動名前設計は「名前に注意を払い、名前とロジックを対応付けること」に大きな価値がある。
名前が重要であり、名前とロジックが対応することに価値がある
名前がプログラム構造を大きく左右する

11.3.2　仕様変更時の「意味の変化」に警戒 
仕様変更によって文脈が変わり、言葉の意味が変化することがある。
そのため、名前は継続的に見直す必要があります。 　

たとえば開発初期に顧客クラスがあり、そのクラスは「個人顧客」を表現したものだったとします。ところがその後の仕様変更により「法人顧客」も扱うようになり、法人に付いて回る登記番号や組織名が顧客クラスに紛れ込んでしまうような状況です。個人顧客なのか法人顧客なのか、顧客クラスの中でロジックが混乱してしまう、ということもありえるわけです。 　
異なる意味が混入しそうな場合は、名前が意味するところを見直します。
名前を変えたり、クラスを分けたりすることで、目的ごとの関心の分離をコードで明確に表現します。
たとえば個人顧客クラスと法人顧客クラスを用意する、といった設計です。 
11.3.3　構造改善を妨げるアンカリング効果 
名前の見直しで注意しなければならないのが、アンカリング効果と呼ばれる心理作用。
アンカリング効果：最初に提示された数値や情報が基準になってしまい、その後の判断を歪めてしまう認知バイアス
たとえば最初に提示された価格が高額だと、後から提示された価格が割安に感じられます。最初の価格が損得の判断基準になってしまうのです。これがアンカリング効果です。アンカリング効果に振り回されないためには、そもそも最初に提示された価格が本当に妥当だったのかどうかを検証する必要があります。 　
アンカリング効果はソフトウェア開発でも発生します。既存のクラス名やメソッド名が基準となってしまい、開発者の判断を歪ませ、混乱させてしまうケースがとても多いのです。本章の商品クラスがその例です。既存の名前に振り回されると、適切な構造を設計できなくなります。 　アンカリング効果は非常に強力かつ厄介です。なぜならアンカリング効果は基準の誤りに気づけない認知バイアスなので、「そもそも名前がおかしい、名前の改善が必要だ」という発想に到達できなくなるのです。なぜ混乱しているのか開発者本人ですら気づいていないケースが大変多く、混乱したままコードを読み書きし続けることになります。
アンカリング効果から抜け出すには、次のような質問を投げかけましょう。 このクラスの正体は何ですか。
実は全然別の概念である可能性がありませんか。 このクラスの目的を教えてください。
また、何をもって目的を達成したとみなすのかも教えてください。 　嘘や欺瞞を見抜く有効な考え方として、「その人が何を言っているかではなく、実際の行動に着目する」という考え方があります。この考え方と同様に、クラスの名前ではなく、クラスが実際に何をやっているのか、それはどんな目的のためにやっているかに着目することで、対象クラスの本当の姿を見抜くことができます。

11.3.4　会話には登場するがコード上に登場しない名前に注意 
会話には登場するがコード上には登場しない名前には注意が必要です。図書館の貸出サービス開発を例に見ていきましょう。次のような会話が、サービス開発では頻繁に発生します。 

会話に登場する重要な概念が、ソースコード上で名前も付けられず、雑多なロジックの中に埋没していることが頻繁に見受けられます。 →詳しい人に聞かないと理解が難しくなり、詳しい人がチームから抜けてしまったら、困難さは一層増すことになる。 
また、こうした「名なしのロジック」はソースコードのあちこちに、無秩序に書き殴られる傾向にあります。なぜなら名前が付いていないと、メソッドとしても、クラスとしても設計されないから既存のソースコード内に、単に仕様通りに動作するようロジックがベタ書きされることになるのです。 　こうなってしまうと、たとえば要注意会員に関して仕様変更が発生したとき、どこからどこまでが要注意会員に関係するロジックなのか、多大な労力をかけて探し回らなければならなくなります。
こうした苦しみから逃れるためには、会話に登場する名前にもっと敏感になる必要がある。そしてその名前を用いたメソッドやクラスを設計することが大事です。 

11.3.5　名前を知らないものは知覚できない 
ジョシュアツリーの法則：名前を知ってはじめて存在を知覚できるようになる、逆に名前を知らないと存在を知覚できないとする認知法則
ドキュメントが残されておらず、詳しい人もチームにいない場合、新しいチームメンバーがなんのクラスやインスタンス変数なのか理解できない。このように、名前を知らないと存在を知覚できないのです。
また、名前を知らないと、機能性にも支障をきたすケースがあります。
ソフトウェア開発では対象とする事業分野の専門知識が必要です。 　たとえば在庫管理システムでは、「安全在庫」や「適正在庫」といった在庫に関する知識が必要です。ECサイトの開発においては、もし「売買契約」という名前も概念も知らないと、商品注文の機能性に支障をきたす可能性が考えられます。 「知らなければ対処できない」というハードルを乗り越えるには、ソフトウェアで解決したい問題、達成したい目的に関して、その分野で使われている言葉を把握し、言葉の意味を理解することが必要。その分野に詳しい人へインタビューしたり、その分野の文献を読むことが役に立つでしょう。 

11.3.6　形容詞で区別しているときはクラス化のチャンス 
見た目では違いがわかりにくコードを同僚に説明する場面で、形容詞で区別して説明しているのを見かけることがよくあります。 このように、意味の異なるものや、条件によって振る舞いが異なるものを、同じ名前・似たような名前で表現しているために違いの見分けが難しくなります。 　チームメンバーが「もともとの～」「補正された～」など形容詞を使って違いを説明する状況がよくあります。この例では詳しい人（先輩社員）がいたために違いがすぐにわかりましたが、詳しい人がチームから抜けてしまうと、周囲のロジックから違いを類推しなければならなくなります。条件が複雑だと、違いの類推は一層困難になります。 　こうした事態を防止するためには、あいまいな命名をせず、意味の違いがわかる命名が大事です。このゲームの例では少なくとも次のように命名すべきです。 
・もともとの最大ヒットポイント：originalMaxHitPoint 
・補正された最大ヒットポイント：correctedMaxHitPoint 　
形容詞を付けてまで違いを表したいものは、それぞれ別のクラスとして設計しましょう。補正された最大ヒットポイントは、もともとの最大ヒットポイントと装備品により決定する関係性があります。この関係をクラスで構造化する。 　もともとの最大ヒットポイントをクラスとして設計し、関連するデータとロジックをカプセル化します。
 OriginalMaxHitPoint {
  private static final int MIN = 10;
  private static final int MAX = 999;
  final int value;

  OriginalMaxHitPoint(final int value) {
    if (value < MIN || MAX < value) {
      throw new IllegalArgumentException();
    }
    this.value = value;
  }
} 　
このように意味の違うものそれぞれをクラスとして設計し構造化することで、各概念の関係性が理解しやすくなります。違いのよくわからないロジックについて同僚と話す場合、形容詞を使った区別に注意しましょう。いくつか例を挙げます。 
・「このフラグが立っているときのUserは要注意会員」 
・「この行のpriceは新品価格で、次の行のpriceは中古価格」 
・「Ticketクラスは、年齢が60歳以上ならシニア料金用になって、さらに平日なら平日のシニア料金用に変わるんだ」 　このように違いを形容詞で表現している場合、それぞれ別のクラスとして設計できないか検討してみましょう。


11.4　意図がわからない名前 
命名で陥りがちなさまざまな悪しきケースと、その対策
意図や目的がわからない名前
計算結果を一時的に格納するためのローカル変数をtmpと番号で命名しています。各変数が何を目的としているのかわかりません。目的駆動名前設計の観点で考えると、関心の分離に貢献しません。 　
あいまいな変数名では、実際の意図に翻訳する作業が読むたびに必要になります。たとえば仕様変更の依頼があったとき、対応するメソッドや変数が何であるのか頭の中で翻訳作業が必要になります。また、チームに新しく参加したメンバーに対しての説明コストが増大します。
翻訳の課題を解決するため、スプレッドシートで用語集や対応表を作成するケースがあります。しかし、この手の資料はメンテナンスがおろそかになりがちです。仕様変更に伴い意味が変わってきたり、説明が抜けたりします。翻訳の時間がかかるということは、当然開発速度の低下を意味します。 　人間の注意力には限界があります。いつでも正確に翻訳できるわけではありません。

11.5.2 DTO(Data Transfer Object)
データクラスは基本的に避けるべきだが、データクラスを使う場面
コマンド・クエリ責任分離(CQRS)と呼ばれるアーキテクチャパターン：更新系と参照系とでモジュールを分離する
参照系はDBから値を取得するだけの処理系で画面表示に使われる
単に取得して表示するだけなので計算やデータの変更は伴わない。この場合、DBの値を格納して表示側に転送するだけのクラスとして設計→DTO
DTO：データ転送用途に使われる設計パターン
値変更の必要がないためインスタンス変数はfinalで宣言し、コンストラクタで値が確定するようにする
